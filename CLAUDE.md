# CLAUDE.md — onboarding for the next Claude session

This file is the canonical context for a Claude (or any human pair-
programmer) opening this repo cold. Read it end-to-end before answering
anything: it covers WHY the code looks the way it does, not just WHAT
it does. The "why" is the part you can't grep for.

---

## 1. What this is

A local-only integration between a **REHAU Nea Smart 2.0** heating /
cooling base station (the white box on the wall, with a small LCD and
an Ethernet/Wi-Fi port) and **Home Assistant**.

The integration ships as a **Home Assistant add-on** built from a
**Node.js monorepo** that lives in this same repo. The repo therefore
has two roles:

- **HA add-on repository** — what you add to HA Supervisor as a
  custom repo (`https://github.com/manuxio/rehau-nea-smart-2-home-assistant`).
  The folder `rehau-bridge/` is the addon HA installs.
- **Source monorepo** — `apps/`, `packages/`, `scripts/`, `tests/`.
  Building it regenerates `rehau-bridge/dist/main.js` and
  `rehau-bridge/web/*` (the addon's published artifacts).

Previous v5.x versions used REHAU's cloud (Playwright + e-mail 2FA +
POP3/OAuth2). **v6.x is a complete rewrite**: nothing leaves the LAN.

Current published version: see `rehau-bridge/config.yaml` → `version:`.

## 4. Tech stack

**Bridge**:
- Node 22+ ESM, TypeScript strict
- Fastify 5 + @fastify/jwt + @fastify/cors + @fastify/static + @fastify/rate-limit + @fastify/swagger(+ui)
- fastify-type-provider-zod (zod schemas → openapi)
- undici (TCP pool; no keep-alive for REHAU)
- cheerio (HTML scrape)
- mqtt.js (LWT + retained + HA discovery)
- pino + pino-pretty
- bcrypt (only native dep; kept as external in tsup; addon container installs it)
- tsup bundles to a single 120 KB `dist/main.js` (ESM, target: node22)
  - **`noExternal: [/^@rehau\//]`** — workspace packages MUST be inlined,
    they don't exist in the addon's node_modules at runtime

**Web**:
- React 18, Vite 5, **`base: "./"`** (so HA ingress prefix works)
- TanStack Query v5 for fetches + optimistic updates
- i18next + react-i18next (it + en; default = en since v0.1.10ish)
- sonner for toasts
- Custom hash router (lib/useHashRoute.ts), no react-router
- TailwindCSS configured but barely used — UI is mostly **inline styles
  with CSS-var colours**. See §6 for the rationale.
- sharp (dev) for the build:icons pipeline

## 5. Build / deploy / dev cycle

### Local dev
```
npm install                                        # workspaces hoist
npm run -w @rehau/bridge dev                       # `node --watch --import tsx src/main.ts`
npm run -w @rehau/web dev                          # Vite on :5173, proxies /api → :8080
```

Why `node --watch` and not `tsx watch`? On Windows `tsx watch` fails
to kill the previous Node process on file change, leaving port 8080
stuck. Node's native watcher is reliable. Plus we have an
`EADDRINUSE` killer at startup in `main.ts` that uses `netstat`/`lsof`
to find and SIGKILL whoever's squatting port 8080 — belt-and-suspenders
for the dev loop. **Disabled in production** (NODE_ENV=production).

### Typecheck / test
```
npm run typecheck --workspaces --if-present
npm run -w @rehau/bridge test                       # vitest, 30 unit tests
```

### Build addon bundle
```
npm run -w @rehau/bridge build                      # tsup → apps/bridge/dist/main.js (~120 KB)
npm run -w @rehau/web build                         # vite → apps/web/dist/
npm run -w @rehau/web build:icons                   # sharp → public/icon-*.png + ha-icon/logo.png
```

### Deploy into the addon folder (manual, scripted later)
Copy `apps/bridge/dist/` → `rehau-bridge/dist/` and
`apps/web/dist/` → `rehau-bridge/web/`, plus
`apps/web/public/ha-icon.png` → `rehau-bridge/icon.png` and
`apps/web/public/ha-logo.png` → `rehau-bridge/logo.png`.

Then bump `rehau-bridge/config.yaml` `version:`. HA shows "Update" on
the addon card based on that version differing from what's installed.

### Test locally without HA
During development I've been deploying via Samba to a live HA instance:
`\\<ha-host>\addons\local-disabled\rehau-bridge\`. After each
edit: build → copy to that folder → bump version in BOTH the samba
copy AND the repo copy → in HA: Reload store → Update → restart addon.
(Alternative: `node --watch` on local apps/bridge with `.env` pointing
to `http://<rehau-base-ip>` — same REHAU device.)

### Publish to GitHub
The repo's `main` branch is protected but the maintainer's push bypasses
the rule (you'll see a "Bypassed rule violations" message — that's fine,
not an error). Use HTTPS, not SSH (SSH key isn't configured on this
machine).

## 6. Code conventions you MUST respect

These were established in real conversations and are NOT obvious from
the code alone:

### Inline styles with CSS vars
- Linter complains "CSS inline styles should not be used". **Ignore
  these warnings everywhere.** This is intentional.
- All colours flow through `var(--accent)`, `var(--bg)`, etc. defined
  in `apps/web/src/index.css` `:root[data-theme]` blocks.
- Theme switching = `<html data-theme="dark|light">`. One source of
  truth. Components don't conditionally pick colours; they just
  reference CSS vars.

### fontSize: ALWAYS rem, never px
- Set globally via `html { font-size: 100% }`.
- Every fontSize in inline styles is a string like `"0.875rem"` (= 14
  px at default OS size).
- This is so iOS/Android system-wide text-size scaling actually scales
  the app.

### Mobile-first PWA
- viewport: `width=device-width, initial-scale=1, maximum-scale=1,
  user-scalable=no, viewport-fit=cover`. Yes the a11y linter screams.
  Confirmed user requirement: app feel beats unlimited pinch zoom.
- `manifest.webmanifest` with `display: standalone`.
- iOS standalone status bar: `apple-mobile-web-app-status-bar-style:
  default` (not `black-translucent` — that overlapped the sticky
  AppHeader with the clock).
- Body has `overflow: hidden`; `#root` is the scrolling container.
  Reason: iOS Safari's native body scroll indicator can't be hidden
  via CSS — the internal `#root` scroller's `::-webkit-scrollbar` CAN.
- `touch-action: pan-x pan-y` on body to block pinch on touch.

### Auth & ingress
- JWT bearer in `Authorization` header. Token lifetime defaults to
  `30d` (was `1h`, raised to keep mobile PWA users logged in).
- Behind HA ingress: `POST /api/v1/auth/ingress` reads the
  `X-Ingress-Path` header HA injects and returns a JWT without
  password. The SPA tries this once on mount before showing the login
  form. See `apps/bridge/src/http/auth.ts` and
  `apps/web/src/lib/auth.tsx`.
- `apiClient.ts` resolves URLs as `new URL("." + path, document.baseURI)`
  so the same bundle works direct (port 8080), under HA ingress
  (`/api/hassio_ingress/<token>/`), and via Vite proxy in dev.
- `onUnauthorized` callback in apiClient triggers logout on 401 (not
  for `/auth/login` or `/auth/ingress` endpoints).

### REHAU device idiosyncrasies (HARD-WON KNOWLEDGE)
- **Connection: close** on every request — REHAU's keep-alive
  implementation hangs. Set in `apps/bridge/src/device/client.ts`.
- **`DEVICE_MIN_GAP_MS`** cool-down between consecutive calls (default
  150 ms) — REHAU's TCP socket table is tiny; back-to-back requests
  exhaust TIME_WAIT and throw `ConnectTimeout`. Raise to 250-400 ms if
  you see saturation in logs.
- **All-or-nothing form semantics**: many installer forms (e.g.
  `room-set-up.html`) require submitting ALL fields, not just the
  changed ones. We always do read → merge → write. See
  `device/source.ts` `setRoomSetup` for the pattern.
- **Single global installer flag**: there's NO cookie. POSTing
  `instPart=<code>` to `/menu.html` flips a process-wide flag on the
  device. EVERY LAN client now sees installer pages until somebody
  GETs `/user-menu.html`. The `InstallerSession` (a process-wide
  mutex) login → fn → unconditional logout pattern serialises this.
  **Exception**: `EXPOSE_IO=true` keeps the session OPEN permanently
  so the I/O page can be polled without thrashing. Documented;
  intentional.
- **°C ↔ °F×10**: REHAU stores temperature as °F×10 in its internal
  setpoints (`setpointHeating`, `HNormH`, etc.). The bridge converts
  on the way in/out. See `apps/bridge/src/device/codecs.ts` and the
  `*H`-suffixed field handling in `device/source.ts`.
- **Per-room fancoil presence**: derived from the presence of the
  `<svg id="FFF">` element on `room-operating.html`. REHAU shows that
  button only when the room's `FanH` setting (on
  `installer-room-set-up.html`) is compatible with the current system
  mode (heating-only / cooling-only / HC). The bridge mirrors what
  the device shows; if the user "doesn't see a fancoil button" in our
  GUI but expects one, the fix is on the device's installer page.
- **Per-room "fancoil running"**: the device emits an extra
  `document.getElementById("FFF").style.fill="#DD0060"` JS line for the
  one room whose fancoil is ACTIVELY running. We grep for that
  literal in the script block — it's the only signal REHAU gives for
  motor state vs configured speed.
- **Language-agnostic parsing** (v6.0.1): REHAU's web UI is in 15
  languages; the user can switch via `user-config-installer.html`. We
  used to match Italian strings ("esterna", "Modulo-U", "Versione")
  and have refactored to structural parsing (single h3, trailing
  digit, label classification by pattern). DON'T re-introduce
  localised text matches.

## 7. MQTT topic structure

```
<base>/<installation-slug>/<device-id>/
├── system/state                 # JSON (Room/System)
├── system/operating_mode/set    # command
├── system/energy_level/set      # command
├── messages, alarms/{active,count}
├── io                           # full IOSnapshot
├── availability                 # "online"|"offline" (LWT)
└── rooms/<room-id>/
    ├── state                    # full Room JSON
    ├── setpoint/set, mode/set, light/set, lock/set, auto_start/set, window_detection/set
```

- `<base>` = `MQTT_BASE_TOPIC` env (default "rehau")
- `<installation-slug>` = slugify(`INSTALLATION_NAME`) — added in v0.1.0
  to allow multiple REHAU installations on one broker without
  collisions
- `<device-id>` = last 8 chars of REHAU `uniqueCode` (or `DEVICE_ID` env)

HA discovery is gated by `MQTT_HA_DISCOVERY=true`. The discovery
"capability signature" in `mqtt/bridge.ts` re-publishes when room
flags or I/O shape changes (so newly-detected fancoils / U-modules
acquire entities without restarting).

## 8. Past decisions / "watch out for"

Things that have come back to bite us. Don't re-discover them:

- **TabBar bottom gap on iOS**: was `padding: 8px 8px 14px`. iOS
  added its safe-area-inset-bottom on top, leaving a big black band.
  Fixed in v6.0.2: `padding: 6px 6px calc(env(safe-area-inset-bottom)
  + 4px)`. Same pattern (env + small constant) applied to the
  SettingsMenu top right button.
- **Sticky AppHeader rides under iOS clock**: fixed by changing
  status-bar-style from `black-translucent` to `default` AND making
  the AppHeader sticky.
- **Optimistic updates "snap back"**: TanStack Query was refetching
  every 5 s during a slow REHAU read-merge-write. Fix: `onMutate`
  cancelQueries + write optimistic; pause the refetchInterval while
  `writing` (any mutation in flight). See RoomDetail.tsx.
- **JWT expiry → cascade of 401 toasts**: apiClient now calls a
  ref-stable `onUnauthorized` callback that triggers a single toast +
  logout. Don't re-introduce the cascade.
- **HA addon doesn't show in Local addons**: usually a Supervisor
  cache. ⋮ → Reload (or Settings → System → Restart → Reload
  Supervisor). The user's HA has the addon nested in
  `addons/local-disabled/rehau-bridge/` because their old custom
  folder was renamed `local-disabled/` and the share root maps to
  `/addons/`. Either name works for scanning — `local-disabled` is
  just user-confusing.
- **HA addon won't see "Update" after changes**: must bump
  `version:` in `config.yaml`. Same-version-no-update is HA behaviour,
  not a bug.
- **`@rehau/types` ERR_MODULE_NOT_FOUND in addon runtime**: tsup keeps
  workspace deps external by default. Use `noExternal: [/^@rehau\//]`.
- **`dotenv` ERR_MODULE_NOT_FOUND in addon runtime**: tsup keeps ALL
  package.json deps external. Solution: addon's `package.json` lists
  every runtime dep (Fastify, mqtt, cheerio, dotenv, …) so `npm
  install --omit=dev` pulls them.
- **CHANGELOG `--` in SVG comment**: librsvg rejects double-hyphens
  in XML comments. Don't put `--` inside `<!-- -->` in SVGs the build
  script processes.
- **i18n typed dictionary**: `as const` on the IT dictionary made TS
  enforce EN strings to match IT literals (impossible). Dropped
  `as const`; `en: typeof it` works structurally.

## 9. Configuration / secrets

- `.env` (gitignored) at repo root for local dev. See `.env.example`
  for the full schema. Notable:
  - `DEVICE_URL=http://<rehau-base-ip>` (the dev REHAU device)
  - `DEVICE_INSTALLER_CODE=<installer-code>`
  - `MQTT_URL=mqtt://<ha-host>:1883`, MQTT_USERNAME, MQTT_PASSWORD (set by your Mosquitto add-on)
  - `INSTALLATION_NAME=Casa` (or whatever; slugified for the topic)
- In the HA add-on, these come from `/data/options.json` via
  `run.sh`. The defaults in `rehau-bridge/config.yaml` `options:` are
  what an end-user sees in the add-on Configuration tab.
- `API_PASSWORD_HASH` is a bcrypt of the resident-UI password.
  Default in addon options is bcrypt of `admin123`. Recommend users
  change it before exposing the addon outside HA ingress.
- `JWT_SECRET` left empty → run.sh generates 48 random bytes once,
  persists to `/data/jwt_secret`, reuses.
